Nom du projet

Authentification vocale biométrique pour Linux/XFCE

Objectif

Ce projet définit et implémente un mécanisme local de connexion Linux basé sur une vérification biométrique vocale combinée à un PIN ou code numérique, tout en conservant le login classique par nom d’utilisateur et mot de passe.

L’environnement cible est Debian/Kali Linux avec XFCE et LightDM.

Le système ne doit pas remplacer de manière destructive la pile d’authentification Linux existante. Il doit ajouter une voie d’authentification optionnelle, testable, désactivable et réversible.

Concept principal

Le flux prévu est :

Écran de login LightDM
  ├── Login Linux classique :
  │     utilisateur + mot de passe
  │
  └── Bouton de login vocal biométrique :
        vérification biométrique vocale
        + challenge ou preuve de présence
        + vérification PIN/code
        = ouverture de session

L’étape vocale doit vérifier l’identité du locuteur, pas seulement transcrire une phrase.

L’étape PIN/code sert de second facteur local.

Le login Linux classique doit rester disponible en permanence.

Plateforme cible

Cible principale :

Système : Kali Linux / Debian
Bureau : XFCE
Display manager : LightDM
Couche d'authentification : PAM
Audio : ALSA / PipeWire / compatibilité PulseAudio
Shell : scripts POSIX compatibles autant que possible
Python : autorisé pour le traitement biométrique

Cible secondaire :

Autres distributions basées sur Debian utilisant LightDM et PAM

Hors périmètre :

Windows
macOS
Authentification cloud-only
Authentification vocale SSH distante
Remplacement complet du display manager

Exigences principales

Exigences fonctionnelles

Le système doit fournir :

  1. Un fallback de login Linux classique par utilisateur/mot de passe.
  2. Une option séparée d’authentification biométrique vocale.
  3. Une vérification locale par PIN ou code numérique après validation vocale.
  4. Un processus d’enrôlement sécurisé pour créer le profil vocal.
  5. Un processus de ré-enrôlement sécurisé.
  6. Un mode test ne modifiant pas le comportement réel du login.
  7. Un mécanisme de désactivation et de rollback.
  8. Des logs utiles sans fuite de secrets.
  9. Des permissions strictes sur les données biométriques et PIN.
  10. Une compatibilité avec les sessions XFCE lancées via LightDM.

Exigences d’authentification

Le chemin d’authentification vocale doit exiger les deux éléments suivants :

correspondance biométrique vocale
ET
correspondance PIN/code

Le système ne doit jamais accepter la voix seule comme authentification complète.

Le système ne doit jamais accepter le PIN/code seul via le chemin de login vocal.

Le login Linux classique reste indépendant et continue d’utiliser le flux PAM/mot de passe existant.

Exigences biométriques vocales

Le sous-système vocal doit supporter la vérification de locuteur indépendante du texte.

Cela signifie :

L'utilisateur peut dire un contenu arbitraire.
Le système vérifie si la voix correspond à l'utilisateur enrôlé.

Cependant, pour limiter les attaques par rejeu, le système devrait supporter un mode challenge optionnel :

Le système affiche un challenge numérique aléatoire.
L'utilisateur lit le challenge à voix haute.
Le système effectue :
  - vérification du locuteur
  - éventuellement vérification du contenu prononcé
  - vérification du PIN/code

Le projet doit distinguer clairement :

speaker verification = qui parle
speech recognition = ce qui est dit

L’objectif central est la vérification du locuteur.

Exigences PIN/code

Le PIN ou code numérique doit :

  1. Être stocké uniquement sous forme de hash.
  2. Ne jamais être stocké en clair.
  3. Ne jamais être journalisé.
  4. Supporter une longueur configurable.
  5. Supporter un délai ou verrouillage après échecs répétés.
  6. Pouvoir être changé par une commande administrative.

Hash recommandé :

Argon2id, yescrypt, bcrypt, ou autre fonction de hachage forte dédiée aux mots de passe

SHA-256 seul n’est pas acceptable pour stocker un PIN.

Exigences PAM

La décision finale d’authentification doit passer par PAM lorsque c’est possible.

Design recommandé :

Greeter LightDM ou helper
  ↓
Service PAM
  ↓
Helper voice-login
  ↓
exit 0 = accepté
exit 1 = refusé

Le projet peut utiliser l’une des approches suivantes :

pam_exec appelant un helper root-owned
module PAM personnalisé
fichier de service PAM dédié

L’intégration PAM ne doit pas casser :

sudo
su
login TTY
SSH
login LightDM classique par mot de passe

Sauf configuration explicite par l’administrateur.

Exigences LightDM/XFCE

L’intégration privilégiée se fait avec LightDM.

Le projet peut fournir :

  1. Un greeter LightDM personnalisé.
  2. Un wrapper autour d’un greeter existant.
  3. Une interface locale séparée lancée depuis l’environnement greeter.
  4. Une documentation pour tester PAM manuellement avant l’intégration graphique.

La session XFCE doit être lancée par le chemin normal du display manager, sans contourner PAM, logind ou la gestion de session.

Éviter les lancements directs de :

startxfce4

sauf en développement/test et clairement marqué comme non recommandé en production.

Modèle de sécurité

Menaces considérées

Le projet doit considérer explicitement :

  1. Attaques par rejeu audio.
  2. Clonage vocal par IA.
  3. Conversion de voix.
  4. Imitation.
  5. Vol des fichiers d’empreinte vocale.
  6. Vol du hash PIN.
  7. Élévation de privilèges locale via helper.
  8. Compromission du greeter.
  9. Micro indisponible ou spoofé.
  10. Déni de service par tentatives répétées.
  11. Verrouillage accidentel de l’utilisateur.
  12. Logs divulguant des secrets.

Règles minimales de sécurité

L’implémentation doit respecter ces règles :

  1. Login par voix seule interdit.
  2. PIN/code obligatoire après validation vocale.
  3. Login classique par mot de passe toujours disponible.
  4. Données biométriques lisibles uniquement par root ou utilisateur de service dédié.
  5. Hash PIN lisible uniquement par root ou utilisateur de service dédié.
  6. Scripts helpers non modifiables par les utilisateurs normaux.
  7. Pas d’injection shell via nom d’utilisateur, chemins audio ou entrées du greeter.
  8. Aucun répertoire runtime world-writable.
  9. Aucun secret en argument de ligne de commande.
  10. Aucun secret dans les logs.
  11. Aucun enrôlement automatique depuis un contexte non authentifié.
  12. Aucune modification destructive de PAM sans sauvegarde.

Layout recommandé

/etc/voice-login/
  config.toml
  users/
    nox/
      voiceprint.embedding
      voiceprint.meta.json
      pin.hash
      policy.toml

/usr/local/libexec/voice-login/
  voice-login-helper
  voice-login-enroll
  voice-login-test
  voice-login-pin-change

/usr/local/bin/
  voice-loginctl

/var/lib/voice-login/
  runtime/
  cache/

/var/log/voice-login/
  voice-login.log
  audit.log

Permissions recommandées :

/etc/voice-login                 root:root 700
/etc/voice-login/users           root:root 700
/etc/voice-login/users/<user>    root:root 700
fichiers voiceprint              root:root 600
fichiers hash PIN                root:root 600
helpers                          root:root 755
configs                          root:root 600 ou 640
logs                             root:adm 640

Capture audio

La capture audio doit :

  1. Utiliser un périphérique micro configuré explicitement quand possible.
  2. Échouer fermée si aucun micro n’est disponible.
  3. Utiliser un sample rate fixe attendu par le modèle.
  4. Stocker l’audio temporaire uniquement dans un répertoire runtime protégé.
  5. Supprimer l’audio temporaire après traitement sauf debug explicite.
  6. Ne jamais conserver l’audio brut par défaut.

Chemin temporaire recommandé :

/run/voice-login/

Permissions :

root:root 700

Enrôlement

L’enrôlement doit être réalisé uniquement après une authentification locale normale.

Il devrait collecter plusieurs échantillons :

5 à 10 échantillons vocaux
3 à 8 secondes chacun
contenu parlé différent
rythme vocal différent

L’enrôlement doit produire :

voiceprint.embedding
voiceprint.meta.json

Le fichier de métadonnées peut contenir :

nom du modèle
version du modèle
sample rate
date de création
nombre d'échantillons
politique de seuil

Il ne doit pas contenir de PIN en clair.

Vérification

La vérification doit :

  1. Capturer un nouvel échantillon vocal.
  2. Extraire un embedding du locuteur.
  3. Le comparer à l’empreinte enrôlée.
  4. Produire un score.
  5. Comparer le score à un seuil configuré.
  6. Exiger la vérification PIN/code après réussite vocale.
  7. Retourner un résultat strict succès/échec.

Le score et le seuil peuvent être journalisés pour debug, mais sans audio sensible ni PIN.

Anti-rejeu et présence

La première implémentation peut être basique, mais le design doit permettre des améliorations anti-rejeu.

Mode challenge recommandé :

1. Générer un challenge numérique aléatoire.
2. Afficher le challenge dans l'interface de login.
3. L'utilisateur lit le challenge à voix haute.
4. Le système vérifie l'identité vocale.
5. Optionnel : vérifier que le contenu prononcé correspond au challenge.
6. L'utilisateur saisit son PIN/code local.

Le challenge doit être aléatoire et courtement valable.

Le système ne doit pas réutiliser d’anciens challenges.

Commandes prévues

Le projet devrait fournir un outil de contrôle :

voice-loginctl

Commandes recommandées :

voice-loginctl enroll --user nox
voice-loginctl verify --user nox --simulate
voice-loginctl test-audio
voice-loginctl set-pin --user nox
voice-loginctl status
voice-loginctl enable
voice-loginctl disable
voice-loginctl rollback
voice-loginctl doctor
voice-loginctl purge

Options obligatoires pour les scripts

Pour les scripts shell, fournir au minimum :

--help
--simulate
--exec
--install
--uninstall
--status
--rollback
--purge
--changelog

Aucune opération destructive ne doit s’exécuter sans flag d’exécution explicite.

Logs

Les logs doivent être écrits sous :

/var/log/voice-login/

Les logs doivent inclure :

timestamp
nom du composant
opération
succès/échec
catégorie de raison

Les logs ne doivent pas contenir :

PIN
phrase prononcée
audio brut
vecteur biométrique complet
hash secret

Exemple de log acceptable :

2026-04-28T22:10:00Z voice-login-helper user=nox action=verify result=failed reason=voice_score_below_threshold score=0.61 threshold=0.72

Comportement en échec

Le chemin voice-login doit échouer fermé.

Exemples d’échec :

micro indisponible
modèle indisponible
configuration invalide
score vocal sous le seuil
PIN incorrect
timeout
voiceprint corrompue
erreur de permissions

Tous ces cas doivent refuser la tentative vocale.

Le login classique par mot de passe doit rester disponible.

Sauvegarde et rollback

Avant toute modification système, l’installateur doit créer des sauvegardes horodatées.

Fichiers potentiellement concernés :

/etc/pam.d/lightdm
/etc/pam.d/lightdm-greeter
/etc/lightdm/lightdm.conf
/etc/lightdm/lightdm-gtk-greeter.conf

Sauvegardes sous :

/var/backups/voice-login/YYYY-MM-DD_HHMMSS/

Le rollback doit restaurer la configuration précédente.

Non-objectifs

Le projet ne doit pas :

  1. Remplacer les mots de passe Linux globalement.
  2. Activer l’authentification vocale SSH par défaut.
  3. Stocker l’audio brut de façon permanente par défaut.
  4. Dépendre de services cloud vocaux.
  5. Utiliser des API en ligne pour le login local.
  6. Modifier les règles firewall.
  7. Modifier des services desktop non liés.
  8. Installer des dépôts non-Debian ou non-Kali sans accord explicite.
  9. Casser l’accès recovery ou TTY.
  10. Masquer les erreurs de sécurité.

Confidentialité

Les données biométriques vocales sont sensibles.

Le projet doit :

  1. Garder les données biométriques en local.
  2. Éviter le cloud.
  3. Éviter toute télémétrie.
  4. Éviter la conservation d’audio brut par défaut.
  5. Documenter exactement ce qui est stocké.
  6. Fournir une commande purge.
  7. Fournir un ré-enrôlement propre.

Packaging

Le projet doit d’abord être installable via scripts locaux.

Packaging futur possible :

paquet Debian
unit systemd pour helper si nécessaire
paquet greeter LightDM

Aucun dépôt externe ne doit être ajouté automatiquement.

Critères d’acceptation

Le projet est acceptable quand :

  1. Le login classique utilisateur/mot de passe fonctionne toujours.
  2. Le voice-login peut être testé sans modifier LightDM.
  3. Le voice-login refuse un mauvais locuteur.
  4. Le voice-login refuse un mauvais PIN.
  5. Le voice-login réussit seulement si voix et PIN passent.
  6. Le voice-login échoue fermé en cas d’erreur audio/modèle/config.
  7. Le rollback restaure PAM/LightDM.
  8. Les logs sont utiles sans fuite de secrets.
  9. Les fichiers biométriques et PIN ont des permissions strictes.
  10. Le projet peut être désactivé sans réinstaller le système.

Choix initial recommandé

Première implémentation recommandée :

Python pour la vérification du locuteur et le PIN
Shell pour les wrappers d'installation/contrôle
PAM via pam_exec pendant le prototype
LightDM seulement après stabilisation CLI et PAM

Première cible recommandée :

Un seul utilisateur local : nox
Une seule machine locale
Session XFCE via LightDM
Pas d'intégration SSH
Aucune dépendance cloud
